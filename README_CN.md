# claude-code-boss-mode

> 不是 Claude Code 的复刻。是给 Claude Code 加一层老板。
>
> 如果 Claude Code 是工程师，这个插件就是老板。

## 这是什么

**Boss Mode** 是一个决策框架插件，在任何工程工作开始之前强制回答三个问题：

1. **这件事值不值得做？** （ROI 关卡）
2. **最小可交付版本是什么？** （范围关卡）
3. **哪些东西明确不做？** （砍需求关卡）

它融合三种老板视角 —— CEO、技术主管、产品经理 —— 形成结构化的评估系统，专治过度设计、范围蔓延和交付拖延。

## 为什么需要这个

Claude Code 是顶级工程师。但工程师优化的是正确性、完备性和优雅。他们不优化交付速度、范围控制和商业结果。

- 工程师提出大规模重构，"顺便做一下"
- 工程师为假想的未来需求构建完美抽象
- 工程师想让每个系统都具备弹性、可扩展、有文档
- 工程师很少问"我们真的需要这个吗？"

老板每天都问这个问题。

## 五个命令

| 命令 | 用途 | 输出 |
|------|------|------|
| `/boss` | 评估一个想法 | SHIP / REVISIT / KILL + 砍掉的部分 |
| `/boss-pr` | 像老板一样审 PR | MERGE / REVISE / CLOSE + 红旗信号 |
| `/boss-plan` | 砍需求、排优先级 | 必做 → 可选 → 砍掉 |
| `/boss-roast` | 吐槽过度设计 | 犀利但可执行的反馈 |
| `/boss-vs-engineer` | 双 Agent 辩论 | 工程师提方案，老板砍方案，协商出最终版 |

## 三种老板视角

### CEO 老板
- 优化目标：**速度、用户价值、市场时机**
- 口头禅："下周能给客户演示吗？"
- 砍掉：基础设施、过早优化、阻碍交付的重构

### 技术主管老板
- 优化目标：**风险、On-call 友好、可回滚**
- 口头禅："凌晨三点这个坏了怎么办？"
- 砍掉：隐式契约、无回滚方案、取巧的捷径

### 产品经理老板
- 优化目标：**范围、时间线、可衡量的结果**
- 口头禅："怎么知道这个有效？哪个指标会动？"
- 砍掉：用户看不到的后端优化、过大的 MVP

## 示例

### 工程师提方案

```
用户：给这个 API 加一层缓存

工程师：我来搭 Redis 方案：
- 连接池
- 自动过期
- 缓存失效策略
- 分布式追踪
- 内存告警
- 指标采集
预计：3 周
```

### 老板介入

```
/boss-pr

老板：REVISE

实际在做的事：
给一个还没证明慢的 API 加 3 周的基础设施。

红旗：
- 没有基准测试证明 API 慢
- 没有缓存命中率监控
- 没有回滚方案
- 过度设计

最小版本：
简单的进程内 LRU 缓存（100 行代码）。先测效果。
如果命中率 >60%，再考虑 Redis。
```

结果：3 周 → 3 天。先衡量，再建设。

## 核心原则

1. **交付胜过完美。** 粗糙的功能在用户手里，比完美的功能在开发中更有价值。
2. **先衡量，再建设。** 别猜，去量。
3. **范围是敌人。** 在范围杀死时间线之前，先杀死范围。
4. **风险是商业决策。** 不是工程问题。
5. **简单才能扩展。** 复杂系统难以运维。从无聊开始。
